# Module 3: Record Manager

## Complexity: ⚠️ Medium
Sequential tasks — each builds on the previous. Core logic is straightforward (hashing + comparison), but touches DB schema, backend service, router, and frontend.

---

## Context

Module 2 built a custom RAG pipeline: upload → extract text → chunk → embed → store in pgvector. The problem: **every upload blindly creates a new document + chunks**, even if the same file was already ingested. Upload `notes.txt` three times = 3 documents, 3x the chunks, degraded retrieval from duplicate results.

Module 3 adds a **Record Manager** — a bookkeeping layer that computes a content hash before ingestion, checks if the content already exists, and determines the correct action: **skip** (identical), **update** (same filename, different content), or **create** (new file).

---

## Task 1: Database Migration

**File:** `backend/migrations/006_record_manager.sql`

- `content_hash` TEXT column on `documents` (SHA-256 of raw file bytes)
- `content_hash` TEXT column on `document_chunks` (SHA-256 of chunk text)
- Unique constraint on `(user_id, file_name)` — one doc per filename per user
- Indexes for fast hash lookups
- Both columns nullable so existing data isn't broken

## Task 2: Record Manager Service

**File:** `backend/app/services/record_manager.py`

- `compute_file_hash(content: bytes)` — SHA-256 of raw file bytes
- `compute_chunk_hash(text: str)` — SHA-256 of chunk text
- `determine_action(file_hash, file_name, user_id, supabase_client)` — returns RecordAction (create/skip/update)
- `diff_chunks(new_chunks, document_id, supabase_client)` — returns (to_add, to_remove_ids, unchanged_ids)

## Task 3: Update Ingestion Service

**File:** `backend/app/services/ingestion.py`

- Import `compute_chunk_hash`, `compute_file_hash`, `diff_chunks` from record_manager
- Add `content_hash` to each chunk row during insertion
- Store `content_hash` on document row when setting status to "ready"
- New function `ingest_document_update()` — uses chunk diffing to only re-embed changed chunks

## Task 4: Update Upload Endpoint

**Files:** `backend/app/routers/files.py`, `backend/app/models/schemas.py`

- Add `content_hash` and `action` fields to `DocumentResponse`
- Upload endpoint computes hash → calls `determine_action()` → handles create/skip/update paths
- Skip returns existing doc with `action: "skipped"` (no background processing)
- Update resets status to pending, triggers `ingest_document_update` in background
- Create proceeds as before with `action: "created"`

## Task 5: Frontend Feedback

**Files:** `frontend/src/lib/api.ts`, `frontend/src/pages/Chat.tsx`

- Add `content_hash` and `action` to `Document` interface
- Upload handler shows toast based on action: info for skipped, success for updated, default for created
- Update path replaces existing file in list; skip path doesn't add duplicate

## Task 6: Tests

**File:** `backend/scripts/test_files.py`

- 6 new dedup tests: first upload (action=created), duplicate upload (action=skipped, same ID), modified upload (action=updated, same ID)
- Cleanup only deletes the test document by ID
