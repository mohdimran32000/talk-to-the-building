# Module 4: Metadata Extraction

## Complexity: ⚠️ Medium
Multiple layers (DB, dynamic schema, extraction service, retrieval, frontend UI) but each step is straightforward. Single-pass feasible if tasks are followed in order.

---

## Context

Module 2 built the RAG pipeline: upload → extract → chunk → embed → pgvector. Retrieval currently returns the top-K most similar chunks across **all** of a user's documents with no way to narrow by document properties. Module 4 adds **LLM-extracted structured metadata** to each document during ingestion, and **metadata filters** to retrieval so users can scope their searches.

**Key design decision:** The metadata schema is **admin-configurable** — stored as JSON in `global_settings`, editable via the settings API (no dedicated UI). The extraction prompt and response schema are built dynamically from whatever fields the admin defines. Ships with sensible defaults.

**Prerequisite:** Module 3 (Record Manager) should be built first — this plan assumes migration `006_record_manager.sql` has been applied.


Each field definition has:
- `name: str` — the metadata key
- `type: str` — one of `text`, `list`, `boolean`, `number`, `date`
- `required: bool` — whether extraction must populate this field
- `description: str` — instruction to the LLM for what to extract

**1c. New RPC with metadata filtering:**
```sql
CREATE OR REPLACE FUNCTION match_document_chunks_with_filters(
  query_embedding  vector(768),
  match_user_id    UUID,
  match_count      INT DEFAULT 5,
  metadata_filter  JSONB DEFAULT NULL
)
RETURNS TABLE (id UUID, document_id UUID, content TEXT, similarity FLOAT)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT dc.id, dc.document_id, dc.content,
         1 - (dc.embedding <=> query_embedding) AS similarity
  FROM document_chunks dc
  JOIN documents d ON dc.document_id = d.id
  WHERE dc.user_id = match_user_id
    AND dc.embedding IS NOT NULL
    AND (metadata_filter IS NULL OR d.metadata @> metadata_filter)
  ORDER BY dc.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;
```

## Task 4: Metadata Extraction Service

**File:** `backend/app/services/metadata.py` (new)

### `build_extraction_prompt(text: str, schema: list[dict]) -> str`
Dynamically builds the extraction prompt from the schema:
```
Analyze the following document and extract metadata for each field.

Fields to extract:
- document_type (text, required): Document category (e.g. report, email, ...)
- topic (text, required): Primary topic in 2-5 words
- entities (list, optional): Key people, organizations, ...
...

Return a JSON object with these exact keys. For required fields, always provide a value.
For optional fields, use null if not applicable.
For "list" type, return an array of strings.
For "boolean" type, return true or false.
For "number" type, return a numeric value.
For "date" type, return an ISO date string (YYYY-MM-DD) or null.

Document text:
{truncated_text}
```

### `build_response_schema(schema: list[dict]) -> dict`
Converts the field definitions into a JSON Schema dict for Gemini's `response_schema` parameter:
- `text` → `{"type": "string"}`
- `list` → `{"type": "array", "items": {"type": "string"}}`
- `boolean` → `{"type": "boolean"}`
- `number` → `{"type": "number"}`
- `date` → `{"type": "string"}` (ISO format)
- Required fields go in the `"required"` array

### `extract_metadata(text: str) -> dict`
1. Read schema via `get_metadata_schema()`
2. Truncate text to first ~4000 words
3. Build prompt and JSON schema dynamically
4. Call Gemini with `response_mime_type="application/json"` and the dynamic `response_schema`
5. Parse response JSON, validate against schema
6. Retry with exponential backoff (reuse `_embed_with_retry` pattern from `ingestion.py`)
7. On failure, return fallback dict with all required fields set to defaults based on type

**Validation:** Call `extract_metadata()` with sample text, verify dict contains keys matching the configured schema.

## Notes

- Existing documents will have `metadata = null` — they won't match any filter but still appear in unfiltered retrieval
- Metadata extraction adds ~1-2s latency to ingestion (one LLM call on truncated text) — runs in background task
- The `@>` JSONB containment operator handles array fields naturally: `{"keywords": ["AI"]}` matches docs whose keywords contain "AI"
- Schema changes only affect **new** ingestions — previously ingested docs keep their old metadata. Users can re-upload to re-extract (Record Manager handles this as an update)
- Supported field types: `text`, `list`, `boolean`, `number`, `date` — the extraction service maps each to the appropriate JSON Schema type for Gemini's structured output
- No dedicated admin UI for schema editing — configure via API or directly in the database
