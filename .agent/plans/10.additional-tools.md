# Module 7: Additional Tools (Text-to-SQL + Web Search)

## Complexity: âš ď¸Ź Medium-High

## Context
The RAG app currently has one tool: `search_documents` (semantic search over chunked text). Module 7 adds two more tools to make the agent more capable:
- **Text-to-SQL**: Excel/CSV data is currently ingested as markdown text chunks â€” fine for semantic search, but bad for quantitative questions ("total sales in Q4", "sum of water consumption"). Text-to-SQL lets the LLM write SQL against the actual structured data.
- **Web Search**: Fallback when the user's documents don't have the answer. Uses Tavily API (purpose-built for RAG).

Both tools integrate into the existing 2-call hybrid flow (non-streaming tool decision â†’ streaming answer with context injection).

## Architecture Decisions
1. **DuckDB for SQL execution** â€” queries run in-memory against user data loaded from JSONB. No dynamic Postgres tables, no RLS complexity, ephemeral and safe.
2. **Tavily for web search** â€” simple SDK, returns structured results with URLs/snippets, built for RAG.
3. **Parallel storage for tabular data** â€” CSV/XLSX still get chunked for semantic search AND stored in `structured_data` table for SQL. Both tools can answer questions about the same file.
4. **Dynamic system prompt** â€” only mentions tools that are enabled, reducing LLM confusion.
5. **Single tool call per turn** â€” keeps the existing flow simple.
6. **All sheets extracted** â€” XLSX files have all sheets extracted, each becomes a separate table in DuckDB.

**Modify:** `backend/app/routers/files.py`, `backend/app/services/ingestion.py`

**Concurrency limiter** â€” Add an `asyncio.Semaphore(3)` to cap concurrent ingestion tasks at 3. Wrap the `background_tasks.add_task()` calls so excess uploads queue instead of all running at once. This prevents 5 simultaneous 50MB uploads from consuming 500MB+ memory.

```python
# files.py â€” module-level
_ingestion_semaphore = asyncio.Semaphore(3)

async def _throttled_ingest(func, *args, **kwargs):
    async with _ingestion_semaphore:
        func(*args, **kwargs)
```

**Note:** `BackgroundTasks` uses sync functions in a thread pool. The semaphore needs to bridge async/sync â€” use `threading.Semaphore(3)` instead of `asyncio.Semaphore` since ingestion functions are sync.

## Task 4: Text-to-SQL Tool Service
**Create:** `backend/app/services/sql_tool.py`

`execute_sql_query(question, user_id, supabase_client) -> str`:
1. Fetch user's `structured_data` from Supabase
2. Build schema description for Gemini (table names + columns)
3. Use Gemini structured output to generate DuckDB SQL
4. Create in-memory DuckDB, load tables from JSONB rows
5. Execute SQL (5s timeout), format result as markdown table (max 50 rows)

Traced with `@traceable(name="query_structured_data", run_type="tool")`.

**Dependencies:** Add `duckdb` to `requirements.txt`

## Task 8: Metadata Enrichment for Tabular Files (Post-build fix)
**Modify:** `backend/app/services/metadata.py`, `backend/app/services/ingestion.py`

**Problem:** XLSX/CSV files extracted via Docling produce raw cell values with no prose context. The metadata extraction LLM couldn't infer meaningful title/topic/etc., so all required fields fell back to `"unknown"`. This means the agentic auto-filter (`search_documents` with metadata filter params) could miss tabular files if the LLM tried to filter by `document_type` or `topic`.

**Fix:** Added `enrich_tabular_text()` in `metadata.py`:
- For CSV: prepends filename + column headers + 5 sample data rows
- For XLSX: prepends filename + per-sheet column headers + 5 sample rows per sheet
- Only activates for `.csv`/`.xlsx`/`.xlsm` extensions; other formats pass through unchanged

Updated `extract_metadata()` signature to accept `file_name` and `file_content` kwargs. Updated both call sites in `ingestion.py` (`ingest_document` and `ingest_document_update`) to pass these through.

**Validate:** Re-upload an XLSX file. Verify metadata fields show meaningful values (not "unknown").
