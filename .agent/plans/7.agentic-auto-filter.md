# Module 4b: Agentic Auto-Filter (Query Understanding)

## Complexity: ✅ Simple
Single backend service + one integration point. No DB changes, no frontend changes. The manual filter bar remains as an override.

---

## Context

Module 4 added metadata extraction during ingestion and manual metadata filters in the UI. Users must manually select filters (document_type, topic, etc.) before sending a message. This module adds an **automatic query-understanding step** that uses the LLM to infer metadata filters from the user's natural language question before retrieval.

**Example:** "What were the rear wheel issues in 2023?" → auto-infers `{"topic": "rear wheel", "publish_date": "2023"}` or relevant keyword/entity filters, then passes them to `retrieve_chunks()`.

**Design decisions:**
- Manual filters take precedence — if the user explicitly sets filters in the UI, skip auto-extraction
- Auto-filter runs only when there are documents with metadata (same guard as existing retrieval)
- Uses a fast, cheap Gemini call with structured output (same pattern as metadata extraction)
- Filters are **loose** — only extract high-confidence filters to avoid over-filtering and missing relevant results
- The auto-extracted filter is logged but not shown to the user (keep UX simple)

## Task 1: Query Filter Extraction Function

**File:** `backend/app/services/metadata.py` (add to existing)

### `extract_query_filters(query: str) -> Optional[dict]`

1. Read schema via `get_metadata_schema()` — same as extraction
2. Build a prompt that asks Gemini to analyze the user's query and extract any metadata filters that would narrow retrieval:
   ```
   Given a user's search query and a metadata schema, extract any metadata filters
   that would help narrow document retrieval. Only extract filters you are confident about.
   If no filters can be inferred, return an empty object.

   Metadata schema fields:
   - document_type (text): Document category (report, email, article, manual, notes, code, other)
   - topic (text): Primary topic in 2-5 words
   - keywords (list): 3-8 keywords for discoverability
   ...

   User query: {query}

   Return a JSON object with only the metadata field names as keys.
   Only include fields where you can confidently infer a filter value from the query.
   For list fields like "keywords", return an array with a single matching term.
   For text fields, return the most likely value.
   For date fields, return YYYY-MM-DD format.
   Omit fields where you cannot determine a value.
   ```
3. Call Gemini with `response_mime_type="application/json"` (no strict response_schema — we want flexibility for optional fields)
4. Parse response, validate keys exist in schema, discard unknown keys
5. Return the filter dict (or None if empty/failed)
6. Wrap in try/except — on any failure, return None (retrieval proceeds unfiltered)

**Key:** Use the same model/API key from settings. Keep prompt concise to minimize latency (~200-500ms).

**Validation:** Call with sample queries:
- "What were the rear wheel issues in 2023?" → should extract topic/keyword filter
- "Show me all technical reports" → should extract `{"document_type": "report", "is_technical": true}`
- "Hello, how are you?" → should return None (no relevant filters)

## Task 2: Integrate into Message Pipeline

**File:** `backend/app/routers/messages.py`

Modify `send_message()` — after the ready docs check, before `retrieve_chunks()`:

```python
# Auto-extract metadata filters from query if no manual filter provided
effective_filter = body.metadata_filter
if not effective_filter and ready_docs.data:
    try:
        from app.services.metadata import extract_query_filters
        effective_filter = extract_query_filters(body.content)
        if effective_filter:
            logger.info(f"Auto-extracted filters: {effective_filter}")
    except Exception as e:
        logger.warning(f"Auto-filter extraction failed (non-fatal): {e}")

context_chunks = retrieve_chunks(
    query=body.content, user_id=user_id, supabase_client=supabase,
    top_k=5, metadata_filter=effective_filter
)
```

**Logic:**
- If `body.metadata_filter` is set (user manually filtered) → use it directly
- If no manual filter and documents exist → call `extract_query_filters()`
- If extraction returns None or empty dict → retrieval is unfiltered (same as before)
- If extraction fails → log warning, proceed unfiltered

**Validation:** Send a message like "What are the rear wheel issues?" without setting any manual filters. Check backend logs for "Auto-extracted filters:" to verify the LLM inferred relevant filters.

## Notes

- No database changes needed — reuses existing `match_document_chunks_with_filters` RPC
- No frontend changes — manual filter bar still works as override, auto-filter is transparent
- Latency impact: ~200-500ms extra per message (one fast Gemini call) — acceptable since it runs in parallel with nothing
- If the auto-filter is too aggressive (over-filtering), users can always clear manual filters or the system falls back gracefully
- The JSONB `@>` containment operator means partial matches work: `{"keywords": ["wheel"]}` matches docs whose keywords array contains "wheel"
