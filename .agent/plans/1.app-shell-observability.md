# Module 1: App Shell + Observability

## Complexity: ⚠️ Medium
Single-pass feasible but involves multiple external services (Supabase, OpenAI, LangSmith) and SSE streaming coordination.

## Context
This is the first module of the Agentic RAG Masterclass. The project is completely fresh - no code exists yet. We're building the foundational app shell: authentication, chat UI with threaded conversations, OpenAI Responses API integration (chat only, no file_search), and LangSmith observability. This sets the stage for Module 2 where we'll replace OpenAI's managed approach with custom retrieval via pgvector.

## Key Decisions
- **No file_search in Module 1** - Focus on auth + chat + streaming. Module 2 builds custom retrieval.
- **OpenAI Responses API** with `previous_response_id` for conversation continuity (not Chat Completions API - that comes in Module 2).
- **Dual storage** - OpenAI manages the conversation chain, but we store messages locally in Postgres for UI display.
- **SSE streaming** - Backend proxies OpenAI's stream to frontend via Server-Sent Events.
- **RLS from day one** - All tables use Row-Level Security.

## Task 3: Database Schema

Run these SQL migrations in Supabase SQL Editor:

### 3a. Threads Table
```sql
CREATE TABLE threads (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title TEXT NOT NULL DEFAULT 'New Chat',
  last_response_id TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX threads_user_id_idx ON threads(user_id);
CREATE INDEX threads_updated_at_idx ON threads(updated_at DESC);

ALTER TABLE threads ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own threads" ON threads FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "Users can insert own threads" ON threads FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "Users can update own threads" ON threads FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY "Users can delete own threads" ON threads FOR DELETE USING (auth.uid() = user_id);
```

### 3b. Messages Table
```sql
CREATE TABLE messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  thread_id UUID NOT NULL REFERENCES threads(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant')),
  content TEXT NOT NULL,
  response_id TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX messages_thread_id_idx ON messages(thread_id);
CREATE INDEX messages_created_at_idx ON messages(created_at);

ALTER TABLE messages ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own messages" ON messages FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "Users can insert own messages" ON messages FOR INSERT WITH CHECK (auth.uid() = user_id);
```

**Validation:**
- Tables visible in Supabase Table Editor with RLS enabled (lock icon)
- Foreign key constraints enforced (message requires valid thread_id)

## Task 5: Backend Chat Endpoints

### 5a. Thread Routes
**File:** `/backend/app/routers/threads.py`

- `POST /api/threads` - Create thread (auth required)
  - Body: `{ "title": "optional string" }`
  - Insert into threads table, return thread object
- `GET /api/threads` - List user's threads (auth required)
  - Query threads ordered by updated_at DESC
- `GET /api/threads/{thread_id}` - Get single thread (auth required)
  - Return 404 if not found (RLS handles ownership)
- `DELETE /api/threads/{thread_id}` - Delete thread (auth required)
  - Cascade deletes messages

### 5b. OpenAI Responses API Client
**File:** `/backend/app/services/openai_client.py`

- Initialize OpenAI client with API key
- `stream_response(content: str, previous_response_id: Optional[str])`:
  - Call OpenAI Responses API with `input` containing user message
  - Pass `previous_response_id` for conversation continuity (if not None)
  - Set `stream=True`
  - Yield text delta chunks as they arrive
  - Return the final `response_id` after stream completes
- Wrap with LangSmith tracing (`@traceable` decorator or manual spans)

### 5c. Message Routes with SSE
**File:** `/backend/app/routers/messages.py`

- `GET /api/threads/{thread_id}/messages` - Get messages for thread (auth required)
  - Return messages ordered by created_at ASC
- `POST /api/threads/{thread_id}/messages` - Send message + stream response (auth required)
  - Body: `{ "content": "user message" }`
  - Steps:
    1. Verify thread exists and belongs to user
    2. Get `thread.last_response_id` for continuity
    3. Insert user message into messages table
    4. Call OpenAI Responses API with streaming
    5. Stream response via SSE: `data: {"type":"token","content":"..."}`
    6. On completion: `data: {"type":"done","response_id":"..."}`
    7. Insert assistant message into messages table
    8. Update `thread.last_response_id` and `thread.updated_at`

### 5d. Pydantic Models
**File:** `/backend/app/models/schemas.py`

- `ThreadCreate(title: Optional[str])`
- `ThreadResponse(id, user_id, title, created_at, updated_at)`
- `MessageCreate(content: str)`
- `MessageResponse(id, thread_id, role, content, created_at)`

**Validation:**
- Create thread via curl with valid JWT, verify in Supabase
- List threads returns only the user's threads
- Send message, receive SSE stream with tokens
- Follow-up message references previous context (proves `previous_response_id` works)
- LangSmith dashboard shows traces for OpenAI calls
