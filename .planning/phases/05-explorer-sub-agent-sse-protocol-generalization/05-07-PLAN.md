**Critical observation for the executor:**

`backend/app/services/sub_agent.py:14` is `from app.services.openai_client import _get_client`. This binds the symbol `_get_client` into the `sub_agent` module's namespace AT IMPORT TIME. When `_section_4_no_progress` does `oc._get_client = lambda: stub_client`, it rebinds the symbol on the `openai_client` module — but `sub_agent.run_explorer_sub_agent` still resolves `_get_client` via its OWN module's namespace, which still points to the original function.

This is the most likely root cause: the test stub is NEVER reached, so `client.models.generate_content(...)` calls REAL Gemini, which returns varying `function_call`s (or no fc at all on some turns), which never triggers the no-progress arm. The "4 tool_starts" observation is consistent with real Gemini iterating a few turns then naturally finishing.

The MINIMAL fix: change the Gemini client resolution in `sub_agent.py` so monkeypatching `oc._get_client` (or `sa._get_client`) reaches the call site. Two equally-defensible options for the executor to choose between, after diagnostic confirmation:
